Guardrails

Three-Layer Safety Model

Input Guardrails (pre-agent)

  • PII scan — redact SSNs, credit card numbers before model call
  • Prompt injection detector — regex + heuristic scan; quarantine on match
  • Size limit — truncate ticket body at 8,000 tokens
  • Encoding normalization — strip HTML, decode MIME, transliterate non-UTF-8

Agent Guardrails (in-loop)

  • can_use_tool permission hooks — enforce confidence thresholds and category hard-stops
  • Max turns — cap at 12 turns; auto-escalate if unresolved
  • Scope enforcement — system prompt explicitly enumerates allowed/disallowed actions

Output Guardrails (post-agent)

  • Decision validator — structured output parsed and validated against schema before write
  • Hard-stop checker — final decision checked against hard-stop list before any write action
  • Audit write — all decisions written to append-only log regardless of outcome
Alternatives Considered

Four Paths Rejected

Alt A — Rejected
Pure Classifier (no agent loop)
Can't enrich with live context (CMDB, HRIS, status page), can't take write actions, can't reason about edge cases. High maintenance overhead for retraining.
Alt B — Rejected
Full LLM, no SDK, direct API
Reinvents tool orchestration, context management, and permission hooks. The Agent SDK provides these primitives production-tested.
Alt C — Rejected
Multi-agent mesh (agent per channel)
State fragmentation, routing inconsistency, harder to audit. Single triage agent with normalized input is simpler and more auditable.
Alt D — Rejected
Human-in-the-loop for every ticket
Doesn't solve the throughput problem. Legal is fine with autonomous action within the defined mandate.
Consequences

Tradeoffs

Benefits

  • Fast path for 61% of ticket volume — password resets, FAQ, duplicate detection
  • Full reasoning trace per decision — explainable to any auditor
  • Permission hooks enforce mandate boundaries at the code level, not just in prompts
  • Subagent pattern keeps security logic isolated and auditable separately

Risks to Watch

  • Agent SDK is relatively new — API surface may evolve
  • Security subagent adds ~2–4s latency on security-flagged tickets
  • Confidence calibration requires ongoing tuning — higher escalation rate in cold-start
  • System prompt length grows as mandate expands — monitor context usage
Open Questions

Three Items for Resolution

Email actions — Do we allow the agent to send emails directly, or only write to the ticketing system?
Current stance: ticketing system only — safer.
Approval queue SLA — What is the SLA for human approvals in the queue?
Proposed: 30 min for P1 · 4 hrs for P2
Multi-language tickets — How do we handle tickets arriving in languages other than English?
MVP: English only · v2: translate-then-process
